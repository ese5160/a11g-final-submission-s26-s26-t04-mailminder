MailMinder:
a secure smart mailbox, built from PCB to cloud.

I built MailMinder as an end-to-end embedded portfolio project on the SiLabs SiWG917: a 4-layer custom PCB, 3D-printed mailbox enclosure, FreeRTOS application, secure MQTT-over-TLS telemetry, Node-RED dashboard, remote unlock, temporary PINs, and HTTP OTAF firmware updates.

What this demonstrates

  • Full-stack embedded ownership across sensing, actuation, network security, dashboard UX, and physical packaging
  • Includes:
    • An IR break-beam across the mail slot for sub-50 ms insertion detection
    • An IMU for tamper / forced-entry detection
    • A magnetic reed switch on the door for open / closed state
    • A 4×4 keypad and TFT display for local PIN entry and live status
    • A servo lock as the secured-door actuator for remote and local unlock flows
  • Enabled with MQTT-over-TLS cloud connectivity, remote unlock, temporary pickup codes, and HTTP OTAF — coordinated by a 6-task FreeRTOS application on the SiLabs SiWG917

Transport

MQTT / TLS

Mailbox events ship over an authenticated TLS link; commands return on the same channel — no polling, no plaintext.

Update path

HTTP-OTAF

A dedicated OTA task accepts a cloud command, downloads firmware from Azure Blob, and resets after the M4 application update completes.

Runtime

FreeRTOS · 6 tasks

Sensor, keypad, actuator, LCD/system control, MQTT, and OTA tasks — coordinated by queues, a status mutex, and explicit event handoff.

Sense. Decide. Act.

A mailbox is a tiny distributed system.

It has to sense a parcel without false-triggering on a draft. It has to decide whether a door movement is a delivery or a break-in. It has to act on a remote unlock within seconds — even when the Wi-Fi just dropped. MailMinder treats each of those as a separate concern, each owned by a single FreeRTOS task, each communicating only through queues and a shared status structure protected by a mutex.

Schematics & layout

From schematic to copper.

The design is partitioned for clarity: a top-level sheet for power and peripheral hookups, and a dedicated SiWG917Y sheet for the MCU itself. The board layout keeps the RF section clear, fans out sensor I/O on the south edge, and routes actuator drive on the east edge.

Hardware

A custom PCBA, designed for one job.

The MailMinder 4-layer PCBA carries the SiWG917 wireless MCU, a USB-C power inlet, an SD-card slot for local storage experiments, and connectors for every peripheral the firmware drives: IR break-beam, IMU, magnetic switch, keypad, LCD, and servo. Boards were brought up from bare copper to bench-tested integration in three weeks.

Bench wiring

From pin map to bench harness.

The wiring plan started as a raw block diagram, then became a traceable bench harness: every sensor, keypad row / column, display line, and servo signal was mapped before the enclosure build.

Power bring-up

Isolate the rail, then trust the rest.

Before sensor or cloud integration, the power path was treated as its own subsystem. We isolated the boost section from the rest of the board, removed test-pad shorts where needed, soldered direct measurement leads, then used a bench supply, electronic load, oscilloscope, and thermal camera to prove startup, discharge, load regulation, and hot spots.

Three moments that make MailMinder real.

01

Hardware that earns its place

Four peripherals, each chosen for one job: an IR break-beam across the mail slot for insertion detection within 50 ms, a 3-axis IMU for tamper, a magnetic switch on the door (≤ 10 mm gap), and a servo with ≥ 60° travel for the lock.

  • HRS-01 — IR break-beam, sub-50 ms detection
  • HRS-02 — Magnetic door, ≤ 10 mm gap
  • HRS-03 — IMU ±2 g, data-ready interrupt
  • HRS-04 — Servo ≥ 60° lock travel

02

Firmware that survives Wi-Fi

The system-control task is the only place state changes. Sensor, keypad, and Wi-Fi tasks publish events into a control queue; the control task publishes commands into an actuator queue and a cloud queue. A small shared status struct — mail_present, door_state, lock_state, network_state, lockout_active — sits behind a mutex and feeds the local display at ≥ 2 Hz.

  • SRS-01/02 — IR break-beam confirms delivery in 2–3 s
  • SRS-04 — Offline PIN unlock within 2 s
  • SRS-05 — Authenticated remote unlock within 5 s
  • SRS-06 — 3 strikes → 30 s keypad lockout
  • SRS-10 — Tamper alert published within 5 s

03

A cloud that talks back

Topics are split by direction. The mailbox publishes delivery, tamper, door_state, lock_state, failed_attempts, and periodic status. The cloud publishes cmd_unlock and cmd_tempcode. Node-RED renders the live state, archives history, and exposes a one-tap remote unlock.

  • SRS-08 — Door open/close events ≤ 5 s
  • SRS-11 — Periodic status at a configurable interval
  • SRS-12 — Authenticated temporary pickup codes
  • OTA — Node-RED command path into a dedicated HTTP OTAF worker

Software architecture

The firmware is more than driver glue.

The current codebase is organized around a state-machine and event-bus model: hardware-facing drivers live under drivers/*, while app/core owns task creation, shared state, MQTT command parsing, cloud publishing, and OTA. Across the application and custom drivers, the project carries roughly 4.7k lines of MailMinder-specific C.

RTOS

Six tasks, clear ownership

sensor_task, user_input_task, actuator_task, system_control_task, mqtt_task, and ota_task each own one timing domain. Shared state is protected by g_state_mutex, while g_cloud_event_queue, g_actuator_queue, and g_ota_queue keep work decoupled.

MQTT

TLS telemetry and commands

The SiWG917 connects to the broker over TLS 1.2 on port 8883. It publishes status, delivery, door, tamper, lock, failed-attempt, and IMU acceleration topics, while subscribing to cmd_unlock, cmd_tempcode, cmd_clear_tamper, and cmd_ota.

STATE

Mailbox behavior as FSMs

The firmware derives mailbox state from IR, reed, and IMU inputs, then latches tamper until a clear command arrives. Temporary codes use a 64-bit timebase for expiry, and cloud unlocks are paired with automatic re-lock behavior.

Mechanical integration

The electronics had to fit the mailbox, not the other way around.

After the electrical and cloud path worked, the design moved into MCAD packaging: a printed enclosure body plus a front mailbox door. The door carries the mail slot and rotating lock geometry, while the body manages the keypad, LCD, PCBA, servo, battery, and cable routing.
